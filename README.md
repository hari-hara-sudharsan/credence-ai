### AI Cannot Execute
The AI engine cannot transfer assets, modify balances, execute settlements, approve loans, change protocol state, mint passports, or update reputation. Only smart contracts can, and only after oracle verification.

### Oracle Verification Layer
The `OracleRegistry` is the trust boundary between off-chain intelligence and on-chain execution. Every signed recommendation includes wallet, score, risk tier, loan parameters, timestamp, nonce, expiry, and signature — validated for authorized signer, signature validity, nonce uniqueness, expiration, replay protection, and payload integrity before execution.

- **Replay protection:** every signature is single-use; reused signatures are rejected and hashes marked consumed.
- **Expiry validation:** signatures carry a short validity window (e.g., 10 minutes) so stale recommendations can't be replayed after conditions change.

### Decision Passport
Every approved decision produces an immutable audit record:
```json
{
  "decisionId": "AI_PASS_001",

```

Credence is a modular financial infrastructure, not a single application. Each contract has one clearly scoped responsibility.

**Protocol stack:**
`Financial Applications (Lending, PayFi, RWA, Insurance, AI Agents) → Credence Trust Layer (7 contracts) → HashKey Chain Mainnet`

**Flow:** User Wallet → Credit Passport → Oracle Verification → Loan Manager → Settlement Manager → Reputation Registry → Updated Financial Identity

| # | Contract | Role | Responsibilities |
|---|---|---|---|
| 1 | **GovernanceRegistry** | Governance layer | Protocol admin, role assignment, permissions, emergency authority (OpenZeppelin AccessControl, least-privilege) |

### Key Controls
- **Role-Based Access Control** via OpenZeppelin AccessControl (e.g. `ADMIN_ROLE`, `ORACLE_ROLE`, `SETTLEMENT_ROLE`, `POOL_ROLE`) — each role scoped to only what it needs.
- **Oracle authentication**: every recommendation must pass signer authorization, nonce, expiry, and ECDSA verification before execution.
- **Replay protection**: nonces make each oracle approval single-use.
- **Signature expiry**: time-bounded validity so stale recommendations can't authorize new transactions.
- **Reentrancy protection** via `ReentrancyGuard` on all value-transferring functions (loan creation, repayment, withdrawals, settlement, liquidity movement).
- **SafeERC20** for standard-compliant, return-value-checked token transfers.
- **Emergency pause** on every core contract for controlled incident response.
- **Input validation** on all financial parameters (limits, values, addresses, IDs, payloads).
- **Contract isolation** — identity, reputation, settlement, lending, governance, verification, and liquidity are all separate contracts, minimizing blast radius.

## Developer Infrastructure & SDK

Developers should build financial products, not rebuild financial trust. Credence exposes trust as a reusable software primitive.

### Quick Start
```bash
npm install @credence/sdk
```
```javascript
import { Credence } from "@credence/sdk";

const profile = await Credence.verify(wallet);
if (profile.trustScore > 850) {
  approveLoan();
}
```

### REST APIs

**Trust Verification** — `GET /api/trust/{wallet}`
```json
{ "wallet": "0x5bb83E60...", "trustScore": 820, "tier": "Institutional", "risk": "Very Low", "passport": true, "loanLimit": 20000, "oracleVerified": true, "lastSettlement": "HSP_202" }
```

**Credit** — `GET /api/credit/{wallet}` → score, maxBorrow, interest, risk, collateralRatio

**Reputation** — `GET /api/reputation/{wallet}` → successfulRepayments, defaults, currentStreak, trustGrowth, lastUpdate

**Financial Identity** — passport status, verification, tier, institutional eligibility

**Oracle Verification** — `GET /api/oracle/{wallet}` → signature, verified, expiry, nonce, oracle name

**Trust Graph** — `GET /api/graph/network` → wallets, trustEvents, reputationUpdates, networkHealth

**Settlement** — `POST /api/hsp/create` → settlement status, verified, receipt, trustImpact
